# ADR-0003: CycloneDX SBOM as Primary Input for Audit Subcommand

## Status

Accepted (2026-03-27)

## Context

The `audit` subcommand needs to evaluate all dependencies in a project for lifecycle health. This requires obtaining a complete list of resolved dependencies (with PURLs) from the target project.

Two broad strategies exist:

- **Per-ecosystem lockfile parsing**: Build or import parsers for each ecosystem's lockfile format (go.mod, package-lock.json, requirements.txt, pom.xml, etc.)
- **SBOM ingestion**: Accept a pre-generated Software Bill of Materials (CycloneDX JSON) that already contains resolved PURLs for all ecosystems

The Go ecosystem offers several open-source lockfile parsing libraries (osv-scanner, trivy, syft), but each presents significant integration challenges detailed in the Alternatives Considered section below.

## Decision

### 1. CycloneDX SBOM as primary input

Accept CycloneDX JSON as the primary input format. SBOMs generated by tools such as syft, trivy, or cdxgen already contain:

- Resolved PURLs for every component (no PURL construction logic needed in uzomuzo)
- Transitive dependencies fully resolved (including nested JARs for Maven)
- A single format covering 20+ ecosystems via the generating tool

A `go.mod` parser is provided as a convenience fallback for Go-only projects, since Go projects are the most common use case within the team. The go.mod parser uses `golang.org/x/mod/modfile` — a lightweight, quasi-stdlib package.

### 2. CycloneDX over SPDX

CycloneDX is the required SBOM format (SPDX is not supported). The reasons are:

- **PURL accessibility**: CycloneDX exposes PURL as a direct field at `components[].purl`. SPDX buries it three levels deep at `packages[].externalRefs[].referenceLocator`, requiring filtering by `referenceType == "purl"`.
- **Stability**: The `purl` field location in CycloneDX has been unchanged across spec versions 1.0 through 1.6.
- **Lightweight dependencies**: CycloneDX JSON parsing uses stdlib `encoding/json` for deserialization and `github.com/package-url/packageurl-go` for PURL normalization — no heavy framework dependencies.

### 3. DependencyParser interface for swappability

A `depparser.DependencyParser` interface is defined in the domain layer. Infrastructure-layer implementations include:

- CycloneDX JSON adapter
- go.mod adapter

Future parsers (npm lockfile, Maven POM, SPDX) can be added as new implementations without modifying the domain or application layers. If osv-scanner's `pkg/lockfile` is ever extracted as a standalone module, it can be adopted as another implementation behind this interface.

## Alternatives Considered

### osv-scanner's pkg/lockfile

Google's [osv-scanner](https://github.com/google/osv-scanner) contains excellent lockfile parsers covering 20+ formats under an Apache-2.0 license. However, `pkg/lockfile` imports `internal/cachedregexp`, an internal package. Go's internal package visibility rules prevent any code outside the `osv-scanner` module from importing packages under `internal/`. This is a Go language constraint, not a quality issue with osv-scanner. If the osv-scanner team extracts `pkg/lockfile` into a separate module in the future, this decision should be revisited.

### trivy as a library

Trivy has lockfile parsers under `pkg/dependency/parser/`. However, importing any single parser pulls in `fanal/types`, which depends on `go-containerregistry` (OCI image types). This adds 1,112 transitive modules to the dependency tree, including Kubernetes client libraries. The uzomuzo module currently has only 3 dependencies; adopting trivy as a library would represent a 370x increase in transitive dependencies. This level of dependency bloat is unacceptable for a CLI tool focused on minimal footprint.

### syft as a library

Syft provides a documented library API (`syft.CatalogPackages()`). However, even importing only the JavaScript cataloger pulls in 215 compile-time dependencies including Docker and containerd libraries (579 total modules). The same dependency bloat concern as trivy applies.

### Building custom lockfile parsers

Writing parsers from scratch for each ecosystem (go.mod, package-lock.json, requirements.txt, Gemfile.lock, etc.) would avoid external dependencies but require significant ongoing maintenance. Each ecosystem has its own resolution semantics, version pinning conventions, and edge cases. The SBOM-first approach delegates this complexity to purpose-built tools (syft, trivy, cdxgen) that already handle it well.

## Consequences

### Positive

- One input format covers all ecosystems with zero ecosystem-specific parsing code in uzomuzo
- No heavy external dependencies added to the module
- Users can generate SBOMs with their preferred tool (syft, trivy, cdxgen) and feed the output to uzomuzo
- The DependencyParser interface keeps the door open for direct lockfile parsing if a lightweight library becomes available

### Negative

- Users must run a separate SBOM generation step before invoking `audit` (extra workflow step)
- CycloneDX-only support means SPDX users must convert their SBOMs (e.g., via `cdx-cli` or regeneration)
- The go.mod fallback covers only Go projects; other single-ecosystem users must still generate an SBOM

### Neutral

- SBOM generation tools (syft, trivy) are already widely adopted in CI/CD pipelines, so the extra step is often already present
- The DependencyParser interface adds minimal abstraction overhead (one interface, two implementations)
