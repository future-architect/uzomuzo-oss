# Dependency Analysis: $ARGUMENTS

Analyze the dependency `$ARGUMENTS` in this project for removal feasibility. Produce a structured report covering ALL of the following sections.

**Scope**: This command is designed for Go projects. The 6-axis evaluation framework is language-agnostic, but the analysis steps below use Go-specific tools (`go mod why`, `go mod graph`, etc.).

## Analysis Steps

1. **Project context**:
   - Check `go.mod` for the project's Go version (determines which stdlib alternatives are available).
   - Run `go mod why -m $ARGUMENTS` to check if removing direct usage will completely eliminate this module, or if it remains as an indirect dependency.
     **Note**: This command can be slow (1-5 minutes) on large projects with cold module caches. If it takes too long, use these faster alternatives:
     - Check if the module is listed as `// indirect` in go.mod
     - Run `go mod graph | grep " $ARGUMENTS@"` to find what depends on it
   - Check the module's license (see the module's GitHub repository or deps.dev page) and whether it depends on CGO.

2. **Import locations**: Find ALL files importing this module.
   - Categorize into production code (`*.go` excluding tests) and test code (`*_test.go`).

3. **Generated code check**: For each file found in Step 2, check for code generation headers:
   - `// Code generated` (standard Go convention, e.g., `// Code generated by MockGen. DO NOT EDIT.`)
   - `//go:generate` directives that reference this module's tools
   Generated files are trivially migrated by re-running the generator with the replacement tool. Focus manual analysis effort on hand-written files only.

4. **Code path classification**: For each usage location, determine:
   - **Active code**: Runs in normal operation (highest priority)
   - **Migration code**: Runs once during upgrade, e.g., database migration functions (lower runtime risk, but supply chain risk remains since the package is in the binary)
   - **Deprecated/dead code**: Behind feature flags or unreachable (lowest priority, consider deletion)

5. **Usage detail & API leakage**: For each hand-written file, identify specific functions, types, methods used.
   - **Critical**: Check if any types from this dependency appear in EXPORTED identifiers. If so, replacing it causes breaking changes.
   - If the dependency is used in more than 20 files, sample 5-10 representative files across different packages. Focus on the "wrapper layer" — files that define wrapper types/interfaces around the dependency (these determine the migration API surface). Summarize usage patterns rather than listing every file.

6. **Code volume**: Count lines of code that directly interact with this dependency. Specify what you counted (import lines, direct API calls, or lines in files that import it).

7. **Transitive dependencies**: Run `go mod graph | grep "^$ARGUMENTS"` to count how many modules this pulls in.

8. **Tool dependencies**: Check if this module provides CLI tools used in the build:
   - Look for `tool` directives in go.mod referencing this module
   - Search for `//go:generate` directives: `grep -r "go:generate.*$ARGUMENTS" --include="*.go"`
   - Check Makefiles and CI scripts: `grep -r "$(basename $ARGUMENTS)" Makefile* .github/ build/`

9. **Maintenance history**: Run `git log --all --oneline --grep="$ARGUMENTS" --since="2023-01-01"` to count historical update PRs.

10. **Replacement feasibility**: Evaluate options (ensuring compatibility with project's Go version):
   - Go standard library
   - Another dependency already in go.mod (consolidation)
   - A smaller/lighter alternative
   - Self-implementation (estimate lines for both logic and tests)

11. **Risk assessment**: Consider:
   - Is this crypto/protocol/security-related? (should NOT self-implement)
   - Is the library actively maintained?
   - Does it have domain-specific complexity?
   - Would self-implementation introduce subtle bugs?
   - Does `go mod why` show it will remain as indirect?

## Output Format

```
## [module name]

### Basic Info
| Item | Value |
|------|-------|
| Project Go version | Go 1.X |
| Full removal possible | Yes / Remains as indirect (via X) |
| License / CGO | MIT / None |
| Files using it | Total: N (prod: N, test: N, generated: N) |
| Code path | Active / Migration-only / Dead code |
| Lines of interaction | N (direct API calls) |
| Transitive deps | N |
| Tool dependencies | mockgen / wire / None |
| Update PRs (2023~) | N |
| Last updated | YYYY-MM-DD |
| Maintenance status | Active / Maintenance / Abandoned |

### Usage Details
- **Primary functions used**: (e.g., `errors.Wrap`, `uuid.New()`)
- **API leakage**: Yes / No (does the dependency's types appear in exported API?)
- **File-by-file breakdown**: (specific functions/types per file, or summary for >20 files)

### Replacement Options
| Option | Estimated effort | Risk |
|--------|-----------------|------|
| (option 1) | N lines | High/Med/Low |
| (option 2) | N lines | High/Med/Low |

### 6-Axis Evaluation

| Axis | Rating | Notes |
|------|--------|-------|
| Update PR reduction | N/yr / No effect | If indirect dep remains, note "direct PRs only" |
| Clean dependency list | High/Med/Low | Does removing direct import clarify intent? |
| Code standardization | High/Med/Low | Can external API be unified with stdlib? |
| Future removal readiness | High/Med/Low | Will this auto-remove when upstream drops it? |
| Supply chain risk | High/Med/Low | Is this abandoned? Does removing reduce attack surface? |
| Code portability | High/Med/Low | Does removing make the code easier to extract/reuse? |

#### Rating anchors
| Axis | High | Medium | Low |
|------|------|--------|-----|
| Update PR reduction | >10 PRs/year eliminated | 3-10 PRs/year | 0-2 PRs/year |
| Clean dependency list | Removes confusing/misleading dep | Removes one of many similar | Marginal clarity improvement |
| Code standardization | Full stdlib replacement available | Partial stdlib, some wrappers needed | No stdlib alternative |
| Supply chain risk | Abandoned + credentials/crypto/auth data passes through | Abandoned + config/metadata passes through | Maintained, or no data passes through (test/CLI/formatter) |
| Code portability | Dep types in exported API (breaking change) | Internal use, moderate coupling | Isolated, easy to swap |
| Future removal readiness | Blocks removal of 3+ other deps | Blocks 1-2 deps | No blocking effect |

### Verdict
- **Recommendation**: Full removal / Self-implement / Submodule isolation / Upstream PR / Keep (wrap) / Test-only (ignore)
- **Effort**: Trivial (<1h) / Small (1-4h) / Medium (1-3 days) / Large (1+ week)
- **Reason**: (1-2 sentences)
- **Priority**: S / A / B / C (S=immediate, A=soon, B=when convenient, C=keep as-is)
- **Cost-effectiveness**: (effort vs. axes summary — e.g., "30 lines of change eliminates a 7-year-abandoned security library from the dependency tree")
- **Note**: Even if indirect dep remains, removing direct usage has independent value (version management delegation, future removal readiness, code portability)
```
